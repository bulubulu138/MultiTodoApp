# Tab/Mode 切换性能优化 - 测试指南

## 测试环境准备

### 1. 启动应用
应用已经在开发模式下启动。如果需要重新启动：
```bash
npm run dev
```

### 2. 打开开发者工具
在应用窗口中按 `Ctrl+Shift+I`（Windows）或 `Cmd+Option+I`（Mac）打开开发者工具，切换到 Console 标签页。

### 3. 准备测试数据
确保应用中有 **300+ 个待办事项**，以便测试大数据量下的性能。

---

## 性能测试步骤

### 测试 1：Tab 切换性能

**目标：** 验证 tab 切换时间 <100ms

**步骤：**
1. 在应用中切换到"待办池" tab
2. 观察控制台输出：`[性能监控] Tab 切换耗时: XXms`
3. 切换到"今日事" tab
4. 再次观察控制台输出
5. 切换到"已完成" tab
6. 再次观察控制台输出

**预期结果：**
- 首次切换：可能 50-100ms（缓存未命中）
- 后续切换回之前的 tab：应该 <50ms（缓存命中）
- 控制台显示：`✅ Tab 切换性能达标 (XXms < 100ms)`

**观察要点：**
- 切换是否感觉瞬间完成？
- 是否有明显的卡顿或延迟？
- 滚动位置是否保持？

---

### 测试 2：视图模式切换性能

**目标：** 验证视图模式切换时间 <150ms

**步骤：**
1. 在"卡片视图"模式下
2. 切换到"专注模式"
3. 观察切换速度和动画流畅度
4. 切换到"紧凑模式"
5. 再次观察
6. 切换回"卡片视图"

**预期结果：**
- 切换应该快速完成（<150ms）
- 动画应该流畅（如果启用）
- 数据应该正确显示

---

### 测试 3：缓存效果验证

**目标：** 验证缓存机制正常工作

**步骤：**
1. 切换到"待办池" tab（首次）
   - 观察控制台：`[性能优化] 排序缓存未命中，重新计算`
   - 观察控制台：`[性能优化] 并列关系分组缓存未命中，重新计算`
   
2. 切换到"今日事" tab
   
3. 切换回"待办池" tab（第二次）
   - 观察控制台：`[性能优化] 排序缓存命中`
   - 切换速度应该明显更快

4. 添加或删除一个待办
   
5. 再次切换 tab
   - 观察控制台：缓存应该失效，重新计算

**预期结果：**
- 首次访问：缓存未命中
- 重复访问：缓存命中，速度更快
- 数据变化后：缓存自动失效

---

### 测试 4：专注模式数据保存

**目标：** 验证异步保存不会丢失数据

**步骤：**
1. 切换到"专注模式"
2. 编辑某个待办的内容
3. **立即**切换到其他 tab（不等待）
4. 切换回原来的 tab
5. 查看待办内容是否保存

**预期结果：**
- 切换时不会被阻塞（立即切换）
- 数据在后台保存成功
- 如果保存失败，会显示警告消息

## 常见问题排查

### 问题 1：切换仍然很慢

**可能原因：**
- 待办数量不够多（<300），优化效果不明显
- 浏览器性能限制
- 其他后台任务占用资源

**排查方法：**
1. 检查控制台是否有错误
2. 查看缓存是否命中
3. 检查性能监控数值

### 问题 2：数据显示不正确

**可能原因：**
- 缓存失效机制有问题
- 数据更新后缓存未清除

**排查方法：**
1. 刷新页面（Ctrl+R）
2. 检查控制台是否有错误
3. 验证缓存 key 是否正确

### 问题 3：专注模式数据丢失

**可能原因：**
- 后台保存失败
- 网络或文件系统问题

**排查方法：**
1. 检查控制台是否有保存失败的警告
2. 检查文件系统权限
3. 尝试手动保存
